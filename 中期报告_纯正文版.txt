与开题内容的一致性方面，课题原计划围绕“数据采集—异常检测—可视化展示—健康建议”展开。当前系统已形成三层联动架构：后端（Spring Boot）、前端（Vue3）与 Python 分析服务（Flask），与开题报告中“多模块协同实现健康监测闭环”的目标一致。

与进度安排的匹配情况方面，按开题报告“2025年12月—2026年3月”的阶段目标，当前已完成数据库与主要业务模块实现、前后端页面搭建、实时链路打通，并进入中期总结与查漏补缺阶段。总体进度基本符合计划，个别细节（如更大规模样本测试与医疗复核流程细化）安排在后续阶段完成。

二、已完成的研究工作及成果

已完成的研究工作如下。

第一，系统架构与工程化搭建完成。已完成后端、前端、Python 服务三端工程搭建，并围绕“可持续开发、可持续联调”目标明确了模块边界。后端基于 Spring Boot 组织控制层、服务层与数据访问层；前端基于 Vue3 完成页面路由、组件划分与接口封装；Python 侧基于 Flask 构建数据分析与结果输出能力。在工程化方面，已梳理三端启动顺序、端口约定与接口职责，减少跨端调试时的定位成本。通过统一项目结构与基础文档沉淀，当前系统已具备后续功能扩展、问题回归与版本迭代的稳定基础。

第二，实时监测核心链路打通。已完成“心率数据→Python 分析→Java 接收→前端实时展示”的端到端流程，并实现多端间的数据格式对齐与时序衔接。Python 服务负责接收或处理心率数据并输出分析结果，后端实时接口负责接收结果并向前端分发，前端监测页面可持续接收并刷新状态。当前前端通过 SSE 机制订阅后端实时推送，能够在不手动刷新的情况下更新监测信息与看板状态。该链路的打通标志着系统已从“静态管理系统”进入“实时感知系统”阶段，为中后期性能压测和稳定性优化提供了可验证基础。

第三，异常检测与数据处理能力实现。当前系统在 Python 侧已经形成了较完整的数据处理与异常识别闭环，不再是单一阈值判断，而是覆盖“原始数据接入—数据质量治理—特征提取—异常判定—结构化输出—后端联动”的完整流程。该流程首先针对实时心率数据天然存在的噪声、抖动、缺失和异常突刺问题，进行了基础过滤和规则约束处理。具体来说，在数据进入判定前，会先剔除明显超出生理合理区间的无效值，处理空值与异常格式，并按照时间顺序重建连续序列，保证后续判断建立在可解释、可追踪的时序数据基础上。对于实时波动较大的场景，系统还加入了基础清洗策略，抑制单点随机抖动对结果的干扰，从源头提高进入判定阶段的数据质量，避免“脏数据直接触发告警”的问题。在特征处理层面，系统不是简单读取一个瞬时心率值，而是围绕连续数据构建判定依据，综合考虑局部变化幅度、短时趋势、持续时长以及异常出现频次等信息，使判断更接近真实业务场景。这样做的核心价值在于，把“偶发波动”和“持续异常”区分开：前者更多作为观察信号，后者才进入告警语义。基于这套处理方式，当前方案能够对连续心率流进行持续监控，并在满足触发条件时稳定输出异常结果，避免只依赖单点阈值造成的误报放大。在异常判定机制上，系统已具备基础规则组合能力，可对“强度异常”和“持续异常”进行区分判定。也就是说，不仅能识别突发异常峰值，还能识别持续偏离基线的慢性风险信号。判定结果会以结构化数据形式输出，字段通常包含人员标识、时间戳、异常类型或等级、触发依据及关键指标等，便于后端系统直接消费。这一点非常关键，因为它使异常检测能力从“模型内部可见”升级为“系统级可用”：后端可据此进行实时推送、记录归档、看板状态更新和后续联动处理，真正形成“可识别、可输出、可联动”的闭环能力。从中期目标达成度来看，当前能力已满足阶段性要求。第一，系统具备稳定识别能力，能在连续监测中发现异常信号；第二，系统具备稳定输出能力，异常结果可标准化、结构化传递；第三，系统具备联动基础，检测结果可直接驱动后端服务和前端展示。更重要的是，这套能力为后续优化留出了清晰路径：通过扩大样本规模、补充不同工况和个体差异数据，可以继续优化误报率与漏报率；通过细化规则分层与阈值自适应策略，可进一步提升模型在真实场景下的鲁棒性与可解释性。因此，当前实现不仅完成了中期“能用”的目标，也构建了后续“更准、更稳”的技术起点。

第四，主要业务功能页面与接口完成。已完成核心业务模块的页面与接口联动开发，覆盖登录、员工管理、疾病信息、心率管理、实时监控、健康建议等主要功能。前端层面已形成相对完整的业务入口与操作流程，支持从基础信息维护到实时状态查看的连续使用路径。后端层面已提供与各业务模块对应的控制器接口，并完成基础参数传递、数据读写与响应返回逻辑，保障前后端交互闭环。通过该阶段建设，系统已具备“管理类功能 + 监测类功能”并行运行能力，可支撑中期演示与后续功能细化。

第五，健康建议生成功能接入。已接入基于 DeepSeek 接口的健康建议生成能力，完成从业务数据组织、提示信息构造到建议结果返回的基本流程。系统可结合员工基础信息、既往疾病信息与心率监测结果生成个性化文本建议，初步体现“监测结果服务于干预建议”的应用价值。当前建议功能已实现可调用、可展示、可复核的中期目标：一方面能在业务页面直接呈现结果，提升系统完整性；另一方面为后续提示词优化、输出结构标准化以及医疗专业复核流程的引入提供可迭代起点。

第六，模型完成路径与前后端接入技术分析。从技术实现看，模型侧工作不是单点算法开发，而是“数据治理+规则判定+服务化输出”三步闭环。第一步在 Python 侧完成心率数据清洗与时序整理，重点处理噪声、缺失值和异常突刺，确保后续判定建立在可追踪的数据基础上；第二步通过连续窗口特征与规则组合完成异常识别，将偶发波动与持续异常分层判定；第三步将判定结果结构化封装并通过 REST 接口输出，使模型能力从“内部可算”转为“系统可用”。在接入链路上，系统采用“Python 分析服务→Java 后端→Vue 前端”的分层架构：Python 负责分析与结果推送，Java 负责鉴权边界、数据汇聚、状态管理与实时分发，前端通过 SSE 持续订阅并渲染结果，从而实现模型与业务系统的低耦合对接。该方案的技术价值在于，模型可独立迭代而不破坏前后端主流程，后续替换算法或细化规则时仅需保持接口契约一致即可完成平滑升级。

第七，前后端页面技术工作分析。前端层面已围绕“管理+监测+干预”主线完成关键页面建设：登录页完成鉴权入口与路由守卫联动；员工管理、疾病信息、心率管理页面实现表格化增删改查、条件检索与状态回显；实时监控页面基于 SSE 构建连续刷新机制并按异常等级进行可视化提示；健康建议页面完成参数组织、请求发起、结果展示与人工复核预留。后端层面针对各页面提供对应控制器与服务能力：统一处理请求参数校验、业务编排、数据库读写、异常响应与结果封装，并为实时场景提供推送接口与状态同步支持。整体上，页面开发不是孤立的 UI 实现，而是以前后端接口契约为核心，通过模块化组件、统一响应结构和可复用服务层实现功能闭环，保证了系统在扩展新页面或新增业务规则时仍具备良好可维护性。

阶段性成果如下：已形成可运行的原型系统，系统已具备后端、前端、Python 分析服务协同运行能力，可按既定流程完成启动、联调与演示，满足中期检查“可运行、可展示”的基本要求；已完成关键模块闭环验证，围绕“监测—识别—展示—建议”主流程完成阶段验证，说明系统各关键环节已由单点实现转向链路化协同；已完成文档化与运行规范沉淀，在项目文档中整理环境要求、启动步骤、关键接口与常见问题，降低团队协作与后期复现实验成本，为论文写作与答辩准备提供支撑。

四、存在的问题与困难

当前主要存在以下问题与困难。

第一，真实高质量样本数据获取难度较大。当前部分验证仍依赖模拟数据或有限样本，虽然能够支持功能联调，但对复杂工况下的泛化能力评估仍不充分。样本规模不足会直接影响异常识别阈值的稳健性，尤其在个体差异显著、环境扰动较强的场景下，模型表现可能出现波动。

第二，跨技术栈联调复杂度较高。Java 后端、Vue 前端、Python 分析服务在接口协议、时间戳处理、异常重试与状态同步方面存在天然耦合，任一环节出现偏差都可能引发链路级问题。当前虽然主流程已打通，但在边界场景（如短时断连、数据突刺、重复推送）下仍需持续优化兼容逻辑与排障机制。

第三，业务规则与医学解释深度有待加强。现阶段异常判定与健康建议已具备工程可用性，但在医学语义严谨性、建议分层颗粒度以及专业复核闭环方面仍有提升空间。若后续不进一步细化规则依据与解释逻辑，可能导致“系统可用但说服力不足”，影响成果在实际场景中的可信度和推广价值。

第四，任务并行推进带来的时间与质量双重压力。后期需要同步完成系统优化、测试验证、论文写作与答辩准备，多线任务并行容易造成精力分散和节奏失衡。若缺少明确优先级与阶段验收标准，可能出现“局部完成较多、整体收口不足”的情况，对最终交付质量形成压力。

针对上述问题，后续将通过“样本扩充+联调清单化+专业复核强化+周度复盘”的方式进行控制。即在数据层面扩大样本覆盖，在工程层面建立问题闭环台账，在业务层面强化医工协同，在管理层面实行里程碑驱动，以降低关键风险并确保后期工作可控推进。
